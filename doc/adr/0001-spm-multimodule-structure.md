## タイトル: モノリシックなターゲット構成から SPM マルチモジュール構成へ移行する

* **承認済**
* stotic-dev
* 日付: 2026-02-21
* 関連 Issue: #75

## 文脈、背景や問題点の説明

homete は Swift ファイル 131 本がメインターゲット `homete` 内にフラットに配置されたモノリシック構成をとっている。変更箇所が少ない場合でもターゲット全体が再コンパイルされるためビルド時間が長く、レイヤー間の依存方向がコードレベルで強制されないため誤った依存が生まれやすい。テストや Preview も同様にターゲット全体に引きずられ重くなっている。

**この構成のまま開発を続けてよいか？**

## 決定事項

* Swift Package Manager（SPM）のローカルパッケージとして Feature ベースの複数モジュールに分割する
* モジュール構成は `HometeDomain` → `HometeUI` → 各 Feature → `homete`（メインターゲット）の4層とし、依存方向を一方向に固定する
* Feature モジュール間の直接依存を禁止し、`HometeDomain` に定義した `RouteResolver` パターン経由で画面遷移を実現する
* `Store` は `HometeDomain` に配置し、メインターゲットで初期化して Environment 経由で各 Feature に注入する
* `CohabitantFeature` は独立したモジュールとせず `HomeFeature` に統合する（同居人管理画面はホーム画面のバリエーションに過ぎないためモジュール数を抑える）
* 段階的移行（Phase 1〜6 の PR 分割）を採用し、1 PR での全移行は行わない

## 考慮した選択肢

* **現状維持（モノリシック構成）**: 移行コストはゼロだが、ビルド時間・依存管理・テスト速度の課題は解決されない
* **Xcode Framework（.framework）ベースのモジュール分割**: バイナリ配布が可能だが、ローカル開発での扱いが煩雑でデバッグしにくい
* **SPM ローカルパッケージによるマルチモジュール（採用）**: Xcode との統合がシームレスで、ソースコードレベルで依存を管理できる。Firebase SDK 等のサードパーティはメインターゲットに留め、SPM のみで完結させる

## 決定結果

### 決定にあたり考慮したメリット

* 変更のあったモジュールのみ再コンパイルされ、ビルド時間が短縮される
* モジュール境界がコンパイラによって強制されるため、誤った依存が混入しにくくなる
* Feature 単体で Preview ビルドが可能になり、UI 開発サイクルが軽量化される
* テストターゲットがメインターゲット全体ではなく必要なモジュールのみに依存するため、ユニットテストの実行が速くなる
* モジュールごとに責務が明確化され、将来のチーム拡大時に変更の衝突が起きにくくなる

### 決定にあたり考慮したデメリット

* 移行作業のコストが大きく、Phase 1〜6 の複数 PR にまたがる継続的な作業が必要
* Swift 6 strict concurrency との整合性（actor 分離・Sendable 等）を各 Phase で都度確認しなければならない
* `RouteResolver` パターンの導入により、画面遷移のコード追跡が間接的になりやすい
